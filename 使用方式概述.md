## 常見使用方式

### 1. 自動補全 (僅限IDE，不消耗點數)
最適合日常開發。

讓補全更準的小技巧：

- 函式、變數命名清楚
- 先寫註解或 TODO
- 先定義輸入/輸出

### 2. Ask
唯讀的模式，個人比較少用，因為通常問完問題就會接著進行修改，不如直接用Agent
適合：

- 解釋一段程式
- 問 API / framework 用法

### 3. Agent
擁有所有工具權限
適合：

- 修 bug
- 寫功能
- 補測試
- 整理文件
- 做 code review

### 4. Plan
複雜任務先不要急著改 code，先請它規劃：

列出
- 要改哪些檔案、怎麼改
- 風險是什麼
- 要怎麼驗收結果

結果比較可控。

## 重要觀念

### 1. 先縮小問題範圍
問問題時要給他參考範圍：
- 指定檔案、片段
- 指定錯誤訊息

### 2. 複雜任務先 plan 再實作
提前審視計畫，避免結果不如意，重複試錯

### 3. 不要把無關問題都塞在同一個 session
這會讓上下文變亂，回答品質下降。

### 4. AI 不一定是對的
關鍵邏輯、資安風險、重要決策，還是要自己確認。

## 哪些功能很值得知道

[Copilot CLI 斜線指令](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference#slash-commands-in-the-interactive-interface)

### 事前準備

#### `/init`
掃描專案，建立全域的 copilot-instructinos.md
執行時建議接上你對專案內容、結構的描述，避免AI盲目猜測

### `/ide`
連接 VS Code(一般會自動連接)

#### `/lsp`
讓 Copilot 更能理解型別、定義與診斷資訊。

### `/add-dir`
新增允許存取的路徑

### 實作功能期間

#### `/model`
切換model

#### `/plan`
先規劃，不直接改程式。

#### `/undo`
撤回上一輪改動

#### `/compact`
把歷史對話濃縮，避免 session 太肥。

#### `/fork`
複製一份當前的對話session，分支探索、嘗試不同結果

### 額外資訊

#### `/context`
看目前對話上下文佔用分佈。

#### `/usage`
看使用量與統計。

#### `/chronicle`
整理工作紀錄、回顧使用習慣。

## 怎麼比較省用量
- 整理好問題及脈絡，一次詢問
- 複雜任務先規劃再做
- 只附必要檔案
- 不要反覆推翻重做
- 不要把和專案無關的問題混在一起

---

## 安全上要注意什麼
- Github Copilot 會讀檔、改檔、跑指令
- 所以要注意目前資料夾是不是可信任
- 看到危險工具權限時不要隨便一路放行
- AI 改完的內容要自己 review

---

## 上手建議
1. 先從 **自動補全 + 問問題** 開始慢慢熟悉
2. 試著指揮他實作功能，輸出結果不如預期，在回頭調整提示詞、Harness
3. 複雜任務用 **Plan + Agent**
4. 改完一定要自己 review
